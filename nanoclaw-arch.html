§2 啟動序列

main() 啟動流程

點擊各步驟查看對應的程式碼與說明。

1
ensureContainerSystemRunning()
確認 Docker / Apple Container runtime 正在執行，並清理上次可能殘留的 orphan containers。
ensureContainerRuntimeRunning(); cleanupOrphans(); // kill nanoclaw-* containers from last run
2
initDatabase() + loadState()
初始化 SQLite，讀取上次的 last_timestamp（message cursor）、last_agent_timestamp（per-group agent cursor）、所有 sessions 和 registered groups。
initDatabase(); // 從 DB 恢復狀態 lastTimestamp = getRouterState('last_timestamp') || ''; lastAgentTimestamp = JSON.parse(getRouterState('last_agent_timestamp')); sessions = getAllSessions(); registeredGroups = getAllRegisteredGroups();
3
startCredentialProxy()
啟動本地 HTTP server，監聽 container 發出的 API 請求，在轉發前注入真實的 API key 或 OAuth token。Container 只知道 ANTHROPIC_BASE_URL=http://host:PORT，永遠不知道真正的 key。
const proxyServer = await startCredentialProxy( CREDENTIAL_PROXY_PORT, PROXY_BIND_HOST // container bridge IP );
4
連線所有 Channels
Channel 在 import './channels/index.js' 時自我註冊。此步驟遍歷 registry，呼叫各 channel 的 factory，credential 缺失時自動跳過。至少一個 channel 連線成功才繼續，否則 fatal exit。
for (const channelName of getRegisteredChannelNames()) { const channel = factory(channelOpts); if (!channel) continue; // credentials missing channels.push(channel); await channel.connect(); }
5
startSchedulerLoop() + startIpcWatcher()
Scheduler 每分鐘掃描 DB，觸發到期的定時任務。IPC watcher 每 500ms 掃描 data/ipc/{group}/，處理 container 回傳的訊息、task 操作、group 註冊等指令。
startSchedulerLoop({ registeredGroups, queue, ... }); startIpcWatcher({ sendMessage, registerGroup, syncGroups, onTasksChanged, ... });
6
recoverPendingMessages() → startMessageLoop()
崩潰恢復：掃描所有 registered groups，若 lastAgentTimestamp 之後有未處理訊息，立即入 queue 重處理。接著啟動永不中止的 poll 主循環。
recoverPendingMessages(); // crash recovery startMessageLoop().catch((err) => { logger.fatal({ err }, 'Message loop crashed'); process.exit(1); });

API Key 模式

Container 帶 x-api-key: placeholder 請求。Proxy 每次都替換為真實 key。

OAuth 模式

Container CLI 用 placeholder Bearer token 向 Anthropic 換取臨時 API key。Proxy 在這個 exchange 請求上注入真實 OAuth token；後續請求用臨時 key，直接通過。

§5 記憶與規則

Agent 如何運用記憶和規則

記憶分為四層，各層的讀寫範圍和存活時間不同。點擊各層查看詳細說明。

① Global CLAUDE.md — 全局規則
read-only 所有 non-main group
▶

存放於 groups/global/CLAUDE.md，掛載為 /workspace/global（read-only）。

Agent runner 在啟動 SDK query 前讀取此檔，將內容 append 到 system prompt：

// agent-runner/src/index.ts const globalClaudeMd = fs.readFileSync('/workspace/global/CLAUDE.md', 'utf-8'); // 傳入 SDK systemPrompt: { type: 'preset', preset: 'claude_code', append: globalClaudeMd }

適合寫入跨 group 通用的行為規則、語氣設定、禁止行為等。Main group 不使用（它有更大的 project 視野）。

② Group CLAUDE.md — 群組專屬規則
read-write 永久持久化
▶

存放於 groups/{folder}/CLAUDE.md，以 /workspace/group（read-write）掛載。

Claude Code SDK 啟動時自動讀取 cwd（即 /workspace/group）下的 CLAUDE.md。Agent 可以直接修改這個檔案來「記住」事情：

cwd: '/workspace/group', // SDK 自動讀取 /workspace/group/CLAUDE.md

這是 Agent「學習」的主要機制——對話中發現的偏好、重要設定，直接寫進這個檔案，下次啟動的 container 仍會記得。

額外掛載目錄（/workspace/extra/*）的 CLAUDE.md 也會透過 additionalDirectories 被讀取：

additionalDirectories: ['/workspace/extra/notes', '/workspace/extra/docs'] // CLAUDE_CODE_ADDITIONAL_DIRECTORIES_CLAUDE_MD=1
③ Session — 對話連續性
read-write 跨 container 啟動
▶

每個 group 有獨立的 data/sessions/{folder}/.claude/，掛載為 /home/node/.claude，存放 Claude Code 的 session transcript。

Session ID 的流動：

// 1. Host 從 SQLite 讀取上次 session ID const sessionId = sessions[group.folder]; // 2. 傳入 container stdin { prompt, sessionId, groupFolder, ... } // 3. agent-runner 用 resume 接續 resume: sessionId, // SDK option // 4. SDK 回傳新 session ID，host 存回 SQLite sessions[group.folder] = output.newSessionId; setSession(group.folder, output.newSessionId);

這讓 Agent 跨多次 container 啟動仍能「記得」前次對話的上下文。

④ Conversations — 對話封存（PreCompact Hook）
write context compaction 前觸發
▶

當 Claude Code SDK 準備進行 context compaction（壓縮對話記憶）時，PreCompact hook 會在壓縮前自動觸發，將完整對話存成 Markdown：

// agent-runner/src/index.ts hooks: { PreCompact: [{ hooks: [createPreCompactHook(assistantName)] }], } // 輸出到： /workspace/group/conversations/{date}-{話題名稱}.md

封存流程：

1. 讀取 transcript JSON → 解析 user/assistant 訊息 2. 查 sessions-index.json 取得 session summary（作為檔名） 3. 格式化成 Markdown，截斷超過 2000 字的訊息 4. 寫入 conversations/{date}-{slug}.md

即使 session 被壓縮，conversations/ 目錄仍保留完整歷史，Agent 可以讀取它們回顧過往對話。

⑤ Container Skills — 指令文件
read-only（每次 sync）
▶

Host 在啟動 container 前，會將 container/skills/* 同步到 data/sessions/{folder}/.claude/skills/，container 內看到 /home/node/.claude/skills/。

每個 skill 是一個帶有 SKILL.md 的目錄，SDK 在收到對應 /指令 時讀取並執行其中的步驟：

container/skills/ agent-browser/SKILL.md ← /agent-browser capabilities/SKILL.md ← /capabilities status/SKILL.md ← /status slack-formatting/SKILL.md

Sync 程式碼（container-runner.ts）：

const skillsSrc = path.join(cwd, 'container', 'skills'); const skillsDst = path.join(groupSessionsDir, 'skills'); // 每次 container 啟動前都 sync，確保最新版本 fs.cpSync(srcDir, dstDir, { recursive: true });

§6 IPC 系統

Container ↔ Host 通訊

Container 透過寫入 JSON 檔案到 /workspace/ipc/ 發出指令，Host 的 IPC watcher 每 500ms 掃描並執行，再刪除檔案。

Host → Container（stdin + IPC input）

初始 prompt 透過 stdin JSON 送入。後續訊息透過 /workspace/ipc/input/ 目錄的 JSON 檔案，或 _close sentinel 結束 session。

// 初始 container.stdin.write(JSON.stringify(input)); container.stdin.end(); // 後續訊息（piping） ipc/input/{timestamp}.json { type: "message", text: "..." } // 結束 ipc/input/_close

Container → Host（IPC output）

Container 將指令寫入 /workspace/ipc/{messages,tasks}/，Host watcher 輪詢處理。身份由目錄路徑決定，不可偽造。

// 發訊息 ipc/messages/{ts}.json { type: "message", chatJid, text } // 操作 task ipc/tasks/{ts}.json { type: "schedule_task", prompt, ... } // stdout（即時串流） ---NANOCLAW_OUTPUT_START--- {"status":"success","result":"回覆文字"} ---NANOCLAW_OUTPUT_END---

支援的 IPC 類型

message
發訊息到指定 chatJid。Non-main group 只能發給自己的 group。
main self only
schedule_task
建立定時任務（cron / interval / once）。
main self only
pause_task / resume_task
暫停或恢復一個已存在的 task。
main self only
cancel_task
永久刪除一個 task。
main self only
update_task
修改 task 的 prompt 或排程設定，自動重算 next_run。
main self only
refresh_groups
觸發 channel sync，重整 available_groups.json 快照。
main only
register_group
將一個 chat JID 註冊為 NanoClaw group。Agent 無法透過 IPC 將 group 設為 isMain（防禦性設計）。
main only

IPC 權限矩陣

操作 Main Group Non-main Group 說明
發訊息給任意 group ✓ ✗ Non-main 只能發給自己的 chatJid
為其他 group 建立 task ✓ ✗ targetFolder !== sourceGroup 直接 block
操作自己的 task ✓ ✓ task.group_folder === sourceGroup
register_group ✓ ✗ 且無法設定 isMain=true
refresh_groups ✓ ✗ 只有 main group 才需要 group 清單
看到 available_groups ✓（全部） ✗（空陣列） writeGroupsSnapshot 的可見性控制

安全設計核心：IPC 目錄路徑決定身份（data/ipc/{groupFolder}/），每個 group 只能寫入自己的 namespace。Host 不相信 JSON 裡宣稱的身份，只看檔案來自哪個目錄。這樣即使 container 被攻破，也無法偽裝成 main group 執行特權操作。
